# 代码编写规范

## 配置
代码设计文档目录: docs/design
计划文档和结果目录: docs/dev
是否提交docs/目录: 是
git commit语言: 中文
文档语言: 中文
注释语言: 中文

## 如何确定下一步的工作
如果用户没有指定，应该思考本系统的最终目标是什么，当前代码状态下的高价值、最紧急的任务是什么。

## 如何做设计
设计命名为 {topic}-{date:YYYYMMDD}-{tag}.md 如，memory-20260606-overview.md

## 如何制定计划
计划文件命名为 PLAN-{XXX:数字}-{tag}.md 放在对应目录，比如命名为: PLAN-001-function.md

## 如何实施代码
按计划实施代码，如果计划实施前有更新，先更新计划文件再实施。实施完代码应该review一遍代码，如果有问题，应该修复，然后生成结果文件，比如PLAN-001-function-OUTCOME.md

## 如何提交代码
遵守我们的配置，如果提交docs/ ，应该在commit消息中记录我们实施的计划文件和结果文件，逐条详细记录我们的变更。commit后要立刻push。

## 如何做接口设计
接口设计首先要区分接口的受众。

【用户接口】面向人、Agent 调用者、业务开发者或最终使用者，核心目标是用低认知负担表达真实意图。它应该贴近用户心智模型，隐藏不必要的实现细节。 从若干用户真实场景出发，倒推用户友好的接口，要求: 简单易用、低认知负担

【内部接口】面向系统模块、运行时、服务、工具、调度器或框架自身，核心目标是保证系统正确性、可维护性、可测试性、可演进性和性能边界。它可以贴近系统实现模型，但必须有清晰的契约、类型、错误语义、状态边界和兼容性规则。

面向用户场景下的接口设计，最好的规范是：用少量贴近现实的概念表达用户真实意图；让简单场景简单，让复杂场景可控，让危险操作显式。

二者不应混用，既要完备，也不应该过度设计。

用户接口不应该被内部实现细节污染；内部接口也不应该为了表面简洁而牺牲精确性、可调试性和系统约束。
