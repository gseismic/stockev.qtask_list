# qtask_list Dashboard 问题处理控制台设计

## 背景

上一版 Dashboard 已经能展示队列、任务状态和手动控制动作，但从真实截图看，它仍然偏“状态面板”：

- 用户能看到很多数字，但需要自己判断哪个问题最重要。
- `ready` 大量积压且没有 Worker 时，页面仍把任务表作为主内容。
- 连接失败时仍可能展示旧数据和危险操作，用户无法判断数据是否可信。
- 投递任务和清空等低频/危险操作占据了与诊断同等的位置。

这类界面的真实目标不是“展示 Redis 队列所有细节”，而是帮助用户快速完成一次运维判断：

1. 这个页面的数据可信吗？
2. 哪个队列最需要处理？
3. 主要问题是什么？
4. 我下一步应该做什么？
5. 如果必须查看单个任务或执行危险动作，入口在哪里？

## 真实使用场景

### 场景 1：队列大量积压但没有 Worker

典型状态：

- `ready > 0`
- `active_workers = 0`
- `dlq = retry = processing = 0`

用户真正要做的是启动消费者，而不是逐条查看 120 个任务样本。因此界面应：

- 在主区域显示明确的问题横幅。
- 把下一步写清楚：启动对应消费者，或确认 Worker 是否部署。
- 任务列表降级为“任务样本”，默认可折叠。
- 不突出 `清空`，避免把危险动作当成常规解决方案。

### 场景 2：存在 DLQ 或 retry

典型状态：

- `dlq > 0` 或 `retry > 0`

用户需要先看失败原因和 payload，再决定单条重试或批量重放。因此界面应：

- 把 DLQ/retry 作为高优先级问题。
- 默认把状态切到有问题的任务集合。
- 批量重放可见但不抢占单条查看入口。

### 场景 3：连接失败或请求失败

典型状态：

- `/api/health` 或后续 API 请求失败
- 页面可能还留着上一次成功加载的数据

用户必须知道这些数据已经不可信。因此界面应：

- 显示全局连接错误横幅。
- 标记数据可能过期。
- 禁用写操作，包括重试、恢复、清空、删除、投递。
- 保留只读数据作为参考，但不能让用户误以为正在看实时状态。

### 场景 4：用户要投递任务

投递是开发调试入口，不是运维首要动作。界面应：

- 默认折叠。
- 放在右侧低优先级位置。
- 连接失败时禁用。

## 信息架构

首屏从上到下：

1. 顶部：连接状态、刷新控制、最后更新时间。
2. 左侧：问题优先排序后的队列列表。
3. 主区域：
   - 当前队列状态标题。
   - 当前队列统计。
   - 问题横幅：显示最重要问题和建议动作。
   - 任务样本：在无 Worker 积压场景下默认折叠；在用户搜索或切换状态时展开。
4. 右侧：
   - 诊断和 Worker 状态。
   - 投递任务折叠入口。
   - 危险操作折叠入口。

## 操作分级

### 常规操作

- 刷新
- 自动刷新开关
- 查看任务详情
- 单条任务重试

### 批量操作

- 重试队列
- 重放死信
- 安全恢复

仅在对应状态有任务且连接正常时可用。

### 危险操作

- 强制恢复
- 清空队列
- 删除任务

危险操作需要更弱的默认曝光，并保留二次确认。

## 非目标

- 不在 Dashboard 中启动业务 Worker 进程，因为 qtask_list 不知道业务项目的启动命令。
- 不引入前端构建链。
- 不改变现有 API 和 Redis 数据结构。

## 验收标准

- 连接失败时有明确全局提示，并禁用写操作。
- `ready > 0 && active_workers = 0` 时，主区域优先提示“无 Worker 导致积压”。
- 大量积压时任务表作为样本展示，不抢占问题处理信息。
- 危险操作不再与常规批量操作并列暴露。
- 当前队列统计口径明确。
- 桌面和移动宽度下无页面级横向溢出。
